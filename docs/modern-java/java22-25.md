# Java 22–25 Features

<DifficultyBadge level="intermediate" />

Java releases every six months. Java 25 (September 2025) is the next LTS after Java 21. These versions finalize several major APIs that were in preview since Java 19–21.

---

## Java 22: Unnamed Variables and Patterns

Finalized in Java 22 (JEP 456). Use `_` for variables you intentionally don't use — no more `ignored`, `unused`, or `e` that you never read.

```java
// Before Java 22 — misleading variable names
try {
    int result = Integer.parseInt(input);
} catch (NumberFormatException e) {   // 'e' never used
    return -1;
}

// Java 22+ — explicit intent: exception is irrelevant
try {
    int result = Integer.parseInt(input);
} catch (NumberFormatException _) {
    return -1;
}

// Unnamed patterns in switch (ignore parts of record structure)
record Point(int x, int y) {}
record Circle(Point center, double radius) {}

Object shape = new Circle(new Point(0, 0), 5.0);

// Only care about the type, not the fields
switch (shape) {
    case Circle(_, double r) -> System.out.println("Circle radius: " + r);
    case _ -> System.out.println("Other shape");
}

// Unnamed loop variable
for (var _ : someList) {
    count++;   // just counting — don't need the element
}

// Unnamed lambda parameter
list.forEach(_ -> count++);  // not available yet — lambdas use (unused) -> instead
```

## Java 22: Foreign Function & Memory API (Finalized)

Replaces JNI and `sun.misc.Unsafe` for calling native code and accessing off-heap memory. Safer, more expressive, and doesn't require writing C headers.

```java
// Before: JNI required C code + native method declarations
// After: pure Java, type-safe, memory-safe

import java.lang.foreign.*;
import java.lang.invoke.*;

public class NativeStrlen {

    public static long strlen(String str) throws Throwable {
        // 1. Get a Linker for the current platform
        Linker linker = Linker.nativeLinker();

        // 2. Find the C standard library
        SymbolLookup stdlib = linker.defaultLookup();

        // 3. Look up the 'strlen' function
        MethodHandle strlen = linker.downcallHandle(
            stdlib.find("strlen").orElseThrow(),
            FunctionDescriptor.of(
                ValueLayout.JAVA_LONG,    // return type: long
                ValueLayout.ADDRESS       // arg: pointer to char
            )
        );

        // 4. Allocate native memory for the string (auto-freed on scope exit)
        try (Arena arena = Arena.ofConfined()) {
            MemorySegment nativeStr = arena.allocateFrom(str);
            return (long) strlen.invoke(nativeStr);
        }
    }
}
```

```java
// Off-heap memory — useful for large data structures avoiding GC pressure
try (Arena arena = Arena.ofConfined()) {
    // Allocate 1 GB off-heap — no GC pressure
    MemorySegment buffer = arena.allocate(1024 * 1024 * 1024L);

    // Write/read structured data
    buffer.setAtIndex(ValueLayout.JAVA_INT, 0, 42);
    int value = buffer.getAtIndex(ValueLayout.JAVA_INT, 0);  // 42
}
// Memory automatically freed when arena closes
```

**Why it matters:**
- Replaces `Unsafe` which is being deprecated (JEP 471/498)
- Used by frameworks like Netty, Chronicle Map, and Panama-based libraries
- Safe: memory is scoped — no dangling pointer issues

## Java 23: ZGC Generational by Default

From Java 23, Generational ZGC (`-XX:+UseZGC -XX:+ZGenerational`) is the default ZGC mode. The non-generational mode is removed in Java 24.

```bash
# Java 21-22: had to opt in
-XX:+UseZGC -XX:+ZGenerational

# Java 23+: just enable ZGC — generational is automatic
-XX:+UseZGC

# Java 24: non-generational ZGC removed
# Old flag -XX:-ZGenerational is now an error
```

Generational ZGC separates young/old generations (like G1) while maintaining sub-millisecond pauses — giving better throughput than original ZGC without sacrificing latency.

## Java 24: Stream Gatherers (Finalized)

Finalized in Java 24 (JEP 485). Custom intermediate stream operations — the long-missing complement to custom `Collector`s.

```java
import java.util.stream.*;
import java.util.stream.Gatherer;

// Before Java 24: couldn't create custom stateful intermediate ops without hacks

// Java 24: Gatherer<T, A, R>
//   T = input element type
//   A = state type
//   R = output element type

// Built-in gatherers (java.util.stream.Gatherers):

// 1. windowFixed — sliding window of fixed size
List<Integer> numbers = List.of(1, 2, 3, 4, 5, 6);

List<List<Integer>> windows = numbers.stream()
    .gather(Gatherers.windowFixed(3))
    .toList();
// [[1, 2, 3], [2, 3, 4], [3, 4, 5], [4, 5, 6]]

// 2. windowSliding — overlapping sliding window
List<List<Integer>> sliding = numbers.stream()
    .gather(Gatherers.windowSliding(3))
    .toList();

// 3. scan — running accumulation (like a stateful map)
List<Integer> runningSum = numbers.stream()
    .gather(Gatherers.scan(() -> 0, Integer::sum))
    .toList();
// [1, 3, 6, 10, 15, 21]

// 4. fold — reduce to single value as a stream element
// 5. mapConcurrent — parallel map preserving order (uses virtual threads!)
List<String> results = urls.stream()
    .gather(Gatherers.mapConcurrent(4, url -> fetch(url)))
    .toList();
```

```java
// Custom Gatherer — deduplicate consecutive equal elements
public static <T> Gatherer<T, ?, T> deduplicateConsecutive() {
    return Gatherer.ofSequential(
        () -> new Object[] { null, false },  // state: [lastSeen, hasLast]
        Gatherer.Integrator.ofGreedy((state, element, downstream) -> {
            if (!(boolean) state[1] || !element.equals(state[0])) {
                state[0] = element;
                state[1] = true;
                downstream.push(element);
            }
            return true;
        })
    );
}

List.of(1, 1, 2, 3, 3, 3, 4).stream()
    .gather(deduplicateConsecutive())
    .toList();
// [1, 2, 3, 4]
```

## Java 24: Virtual Threads — Synchronize Without Pinning

Java 24 (JEP 491) fixes the biggest virtual thread limitation: pinning.

### The Pinning Problem (Before Java 24)

```java
// Before Java 24: synchronized blocks PINNED virtual threads to platform threads
// Pinning = virtual thread can't unmount from platform thread while blocked

public synchronized void processRequest() {
    var result = database.query(sql);  // blocks here — platform thread is PINNED
    // Other virtual threads can't use this platform thread while waiting
    // → Defeat the purpose of virtual threads under high concurrency
}
```

```
Before Java 24 (pinned):
Platform Thread 1: [VT-1 waiting for DB] ← PINNED, useless
Platform Thread 2: [VT-2 waiting for DB] ← PINNED, useless
Platform Thread 3: [VT-3 processing]

After Java 24 (no pinning):
Platform Thread 1: [VT-3] → [VT-7] → [VT-12] ...  ← carrier for any VT
Platform Thread 2: [VT-4] → [VT-8] → [VT-15] ...
Platform Thread 3: [VT-5] → [VT-9] → [VT-11] ...
```

```java
// Java 24+: synchronized blocks no longer pin virtual threads
// Existing code with synchronized methods works correctly with virtual threads
// No code changes needed — just upgrade to Java 24

public synchronized void processRequest() {
    var result = database.query(sql);  // virtual thread unmounts, platform thread free
}
```

**Impact:** Legacy synchronized code now works efficiently with virtual threads. You no longer need to replace `synchronized` with `ReentrantLock` to avoid pinning.

## Java 24: AOT Class Loading & Linking

JEP 483. The JVM can cache class-loading and linking work from a previous run, making subsequent startups significantly faster — without GraalVM native image.

```bash
# Step 1: training run (creates the AOT cache)
java -XX:AOTMode=record -XX:AOTConfiguration=app.aotconf -jar myapp.jar

# Step 2: production run (uses the cache)
java -XX:AOTMode=on -XX:AOTConfiguration=app.aotconf -jar myapp.jar
```

```
Without AOT cache:  startup ~3–5 seconds (typical Spring Boot)
With AOT cache:     startup ~1–2 seconds (classes already loaded/linked)
GraalVM native:     startup ~50ms (but long build, reflection restrictions)

AOT cache = middle ground: easy to use, significant speedup, no build-time restrictions
```

## Java 24: Quantum-Resistant Algorithms

JEP 496 (ML-KEM) and JEP 497 (ML-DSA). NIST-standardized post-quantum cryptographic algorithms now in the JDK.

```java
// ML-KEM (Module-Lattice Key Encapsulation) — replaces RSA/ECDH for key exchange
KeyPairGenerator kpg = KeyPairGenerator.getInstance("ML-KEM");
kpg.initialize(new NamedParameterSpec("ML-KEM-768"));  // or ML-KEM-512, ML-KEM-1024
KeyPair kp = kpg.generateKeyPair();

// ML-DSA (Module-Lattice Digital Signature Algorithm) — replaces ECDSA for signatures
KeyPairGenerator sigKpg = KeyPairGenerator.getInstance("ML-DSA");
sigKpg.initialize(new NamedParameterSpec("ML-DSA-65"));
KeyPair sigKp = sigKpg.generateKeyPair();

Signature sig = Signature.getInstance("ML-DSA");
sig.initSign(sigKp.getPrivate());
sig.update(data);
byte[] signature = sig.sign();
```

**Why it matters:** Quantum computers can break RSA and ECDSA. These algorithms are resistant to quantum attacks — important for TLS, code signing, and long-lived secrets.

## Java 25 Preview: Structured Concurrency (Likely Final)

Available as preview since Java 21. Expected to finalize in Java 25.

```java
// Run subtasks as a structured unit — all succeed or all cancel together
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {

    Subtask<User>    user    = scope.fork(() -> userService.fetch(userId));
    Subtask<Product> product = scope.fork(() -> productService.fetch(productId));

    scope.join()           // wait for all subtasks
         .throwIfFailed(); // propagate first failure

    // Both succeeded — safe to access results
    return new Page(user.get(), product.get());
}
// Scope closes: all subtasks guaranteed complete or cancelled
```

**vs CompletableFuture:**
- Subtask lifetime is bounded by the scope — no leaked threads
- Cancellation propagates automatically on failure
- Stack traces are coherent — structured like synchronous code

## Java 25 Preview: Scoped Values (Likely Final)

Available as preview since Java 20. Replaces `ThreadLocal` for virtual thread contexts.

```java
// ThreadLocal problem: leaks across thread pool reuse, expensive for virtual threads
// ScopedValue: immutable, bounded lifetime, efficient for millions of virtual threads

public class RequestContext {
    public static final ScopedValue<User> CURRENT_USER = ScopedValue.newInstance();
    public static final ScopedValue<String> REQUEST_ID = ScopedValue.newInstance();
}

// Set values for the duration of a request
ScopedValue.runWhere(
    RequestContext.CURRENT_USER, authenticatedUser,
    () -> {
        // CURRENT_USER is bound for the entire call tree within this scope
        orderService.placeOrder(items);   // can access CURRENT_USER
        auditService.log();               // can access CURRENT_USER
        // value is automatically unbound on scope exit
    }
);

// Read from anywhere in the call tree (no passing through method parameters)
public void placeOrder(List<Item> items) {
    User user = RequestContext.CURRENT_USER.get();  // always available in scope
}
```

## Interview Quick-Fire

**Q: What are Stream Gatherers and how do they differ from Collectors?**
Gatherers (Java 24) are custom *intermediate* stream operations — placed between source and terminal op. `Collectors` are terminal: they fold the stream into a result. Gatherers can be stateful (e.g., windowing, running sums, deduplication) and can produce more, fewer, or the same number of elements as their input. Built-ins include `Gatherers.windowFixed()`, `windowSliding()`, `scan()`, and `mapConcurrent()`.

**Q: What was the virtual thread pinning problem and how was it fixed?**
Before Java 24, a virtual thread entering a `synchronized` block while blocking on I/O was *pinned* — it couldn't unmount from its carrier platform thread, preventing other virtual threads from using that thread. This defeated virtual threads' concurrency benefits in code using `synchronized`. Java 24 (JEP 491) made synchronized blocks non-pinning — virtual threads now correctly unmount while blocked inside synchronized methods, so existing legacy code works efficiently without rewriting to `ReentrantLock`.

**Q: What is the Foreign Function & Memory API replacing?**
It replaces two legacy mechanisms: JNI (Java Native Interface) for calling native C/C++ functions, and `sun.misc.Unsafe` for off-heap memory access. The FFM API (finalized Java 22) is safe — memory is scoped via `Arena` objects, preventing leaks and dangling pointers. It also eliminates the need to write C headers and shared libraries just to call a native function.
